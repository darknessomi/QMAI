AI大纲会话窗口交互重构设计

主题：意图分析与渐进式阶段展示
日期：2026-07-09
方案：方案 B — 显式意图闸门阶段 + 独立阶段状态机

背景与问题

现有实现的三个核心问题

  1. 直接生成，缺少意图分析：用户点击「章节细纲」等模块按钮后，handleGenerateSection 直接构造生成 prompt 发送给 Agent，没有「用户到底要生成哪些章节的细纲」这一意图分析步骤。任务理解完全交给 LLM 在 prompt 文字中自行完成。
  2. 思考过程展示简陋：大纲面板使用 OutlineThinkingBlock 显示原始流式文本块，没有阶段划分。固定 6 阶段面板 AgentWorkflowPanel 未接入大纲面板，且是预先全部显示而非渐进式出现。
  3. 流程无连贯性与推荐：无阶段间数据传递机制；生成完成后无「下一步推荐」，用户需自行决定后续操作。

设计目标

  • 点击模块按钮后先做意图清晰度分析，明确时全自动推进，不明确时智能推荐选项供用户选择
  • 思考阶段改为事件驱动渐进式出现，未激活阶段不显示，已完成阶段折叠为摘要
  • 生成阶段间自动推进、数据有效流转，AI 自动选择技能并提取资源
  • 一章一个独立文件，生成完成后推荐下一步（仅限大纲体系内，严禁推荐正文生成）

关键设计决策

1. 不改动共享文件：workflow-trace.ts 和 agent-workflow-panel.tsx 被 AI 会话面板共用，改动会影响其他功能。所有新逻辑在 outline 专属文件中实现。
2. 意图清晰度通过结构化输出强制：AI 必须先输出 intent_clarity 结构化 JSON，前端解析后决定是否暂停等待用户输入。不靠 AI 自觉，与项目教训一致。
3. 复用现有 Agent 循环：仍走 AgentRunner.run，工具注册不变（route_task / apply_skill / read_outline / read_chapter / write_outline_node 等），在外围增加意图分析层和渐进式渲染层。

意图清晰度分析层

触发时机

用户点击「章节细纲」等模块按钮时，不再直接构造生成 prompt，而是构造意图分析 prompt，让 AI 先做意图清晰度判断。

意图清晰度判定规则

AI 读取已有资料后，根据模块类型判断意图清晰度：

模块类型判定依据needs_input 触发条件
章节细纲已有章节列表 vs 已有细纲文件完全没有章节 / 大范围缺细纲需用户缩小范围
人物小传已有角色列表 vs 已有小传文件角色来源不明确（总纲提及但未细化）
组织势力当前卷涉及的势力 vs 已有势力设定卷范围不明确
力量体系总纲中的体系设定 vs 已有文件体系框架未在总纲中定义
其他模块类似策略信息不足时

意图分析输出格式

AI 输出需包裹在 <!-- intent_clarity --> 标记中：

<!-- intent_clarity -->
{
  "clarity": "clear" | "needs_input",
  "module": "章节细纲",
  "analysis": "检测到共有35章，其中0章有细纲",
  "detectedScope": "clear时填写范围描述",
  "options": [
    {"id":"A","label":"生成前面缺失的细纲","description":"为第1-35章逐一生成细纲"},
    {"id":"B","label":"根据已有章节内容分析后生成后续细纲","description":"读取现有正文，推断后续章节走向并生成细纲"},
    {"id":"C","label":"生成最近5-10章的细纲","description":"为第26-35章生成细纲"},
    {"id":"D","label":"自定义","description":"由你描述要生成的内容范围或故事方向"}
  ],
  "question": "needs_input时的自然语言提问，clear时为空"
}
<!-- /intent_clarity -->
输出约束：clear 时只输出上述 JSON，不生成正文。needs_input 时输出 JSON 后用自然语言在会话中提出澄清问题 + 推荐选项。

前端解析与控制流

handleGenerateSection(title, requestHint)
  → handleSend(意图分析prompt, { phase: "intent_analysis" })
  → Agent 执行（调用 list/read 工具收集信息）
  → AI 输出 intent_clarity JSON
  → 流结束后，parseIntentClarity(lastMessage.content)
      → clear:  自动 handleSend(生成prompt, { phase: "generation", scope: result })
      → needs_input: 渲染 AI 的自然语言提问 + 推荐选项到会话
                     设置消息状态为 waiting_user_input
                     用户点选选项或自由输入
                     → handleSend(用户选择/回复, { phase: "generation", scope: 上次result })
                     → 直接注入生成，不再二次分析
防无限循环：用户回复后直接注入生成 prompt，不再做二次意图分析。避免 AI-用户-AI 的无限循环。

智能推荐选项策略

当意图不明确（needs_input）时，AI 根据分析结果提供 4 类推荐选项：

  1. 全部缺失项生成：如「生成前面缺失的细纲」
  2. 基于已有内容推断：如「根据已有章节内容分析后生成后续细纲」
  3. 最近范围生成：如「生成最近 5-10 章的细纲」
  4. 自定义：用户自行描述要生成的内容范围或故事方向

状态机扩展

在 outline-workflow-state.ts 新增 intent_analysis 和 waiting_user_input 阶段：

idle → intent_analysis → sufficiency_check           (clear)
                        → waiting_user_input → sufficiency_check  (用户回复后)

渐进式思考阶段展示

现状问题

大纲面板用 OutlineThinkingBlock 显示原始流式文本块，无阶段划分。固定 6 阶段面板 AgentWorkflowPanel 未接入大纲面板。

新组件：OutlineWorkflowStages

在 outline 场景内新建独立组件，不改动共享文件。

┌─ 思考过程 ─────────────────────────────┐
│  ✓ 任务理解 · 识别为生成大纲意图 · 1.2s │ ← 折叠摘要，点击展开
│  ✓ 范围分析 · 检测到3章缺失细纲 · 0.8s │ ← 折叠摘要
│  ▶ 技能选择 · 正在选择写作技能...      │ ← 当前阶段，展开显示
│     └ 大纲生成技能已加载               │
│                                        │
│  （后续阶段尚未出现，不显示）           │
└────────────────────────────────────────┘

阶段定义（7 个阶段）

序号阶段激活条件完成条件数据来源
1任务理解检测到 route_task 调用route_task 返回结果agentToolCalls 中的 route_task
2范围分析检测到 intent_clarity 输出intent_clarity 解析完成content 中的 intent_clarity 块
3技能选择检测到 apply_skill 调用apply_skill 返回结果agentToolCalls 中的 apply_skill
4上下文准备检测到 list/read 工具调用所有 read 调用完成agentToolCalls 中的 list_*/read_*
5工具调用检测到 write/其他工具调用工具调用完成agentToolCalls 中的 write_*/其他
6思考与角色检测到 thinking 块流式 thinking 结束separateThinking(content).thinking
7生成与校验检测到正文输出开始质量检查完成separateThinking(content).answer + outlineSaveRequest

渐进式出现机制

核心原则：只有 status 不为 hidden 的阶段才渲染。阶段初始全部为 hidden，当检测到对应事件时变为 active，下一阶段出现时当前阶段变为 done 并折叠。 
type StageRenderStatus = "hidden" | "active" | "done"

// 阶段初始全部为 hidden
// 检测到对应事件 → 该阶段变为 active
// 下一阶段出现 → 当前阶段变为 done 并折叠
const stages = buildOutlineStages(agentEvents)
const visibleStages = stages.filter(s => s.status !== "hidden")

折叠/展开行为

  • 默认行为：done 阶段折叠为单行摘要，active 阶段展开显示完整内容
  • 用户操作：点击任意阶段行可手动 toggle 展开/折叠
  • 过渡动画：新阶段出现时 opacity 0→1 + translateY(4px→0)，250ms ease-out

已完成阶段摘要格式

✓ 任务理解 · {意图标签}（置信度{百分比}） · {耗时}
✓ 范围分析 · {分析摘要} · {耗时}
✓ 技能选择 · {技能名} · {耗时}
✓ 上下文准备 · 读取了{N}个文件 · {耗时}
✓ 工具调用 · 调用了{N}个工具 · {耗时}
✓ 思考与角色 · {思考摘要} · {耗时}
✓ 生成与校验 · 生成{N}个文件 · {耗时}

推荐策略表（仅限大纲体系内）

完成模块推荐方向
章节细纲人物小传（涉及未立传角色）/ 组织势力设定（涉及新势力）/ 伏笔计划（埋设伏笔）
人物小传组织势力设定（角色所属势力）/ 力量体系（角色能力归属）
组织势力力量体系（势力能力层次）/ 地理设定（势力分布区域）
力量体系人物小传（能力归属角色）/ 背景设定（体系运转规则）
伏笔计划章节细纲（伏笔融入章节）/ 人物小传（伏笔相关角色）

推荐内容由 AI 根据实际生成内容动态推断，读取刚生成的文件内容，分析其中提及但尚未建立的关联项，给出针对性推荐。用户点击推荐后自动构造对应模块的生成 prompt 并发送；选择「自定义」时聚焦输入框等待用户输入。

关键约束汇总

  1. AI 大纲面板严禁生成正文，推荐中严禁推荐「生成对应章节正文」
  2. 用户回复后直接注入生成，不再二次意图分析，避免无限循环
  3. 批量保存支持部分取消，用户可取消勾选其中部分文件
  4. 不改动共享文件（workflow-trace.ts / agent-workflow-panel.tsx / chat-panel.tsx / plan-execute-policy.ts / task-router.ts）
  5. 意图清晰度通过结构化输出强制，不靠 AI 自觉
  6. 阶段渐进式出现，未激活阶段不显示，已完成折叠为摘要

测试与验证要求

  1. 意图分析测试：点击章节细纲，验证 AI 是否输出 intent_clarity JSON，前端是否正确解析 clear / needs_input
  2. 渐进式阶段测试：验证 7 个阶段按事件驱动出现，未激活阶段不显示，已完成阶段折叠为摘要
  3. 文件输出测试：验证每章/每项生成独立文件，目录结构正确
  4. 批量保存测试：验证批量文件列表展示，部分取消功能正常
  5. 下一步推荐测试：验证推荐卡片渲染，点击后正确构造对应模块 prompt；验证推荐中不含正文生成选项
  6. 回归测试：验证 AI 会话面板的思考过程展示不受影响（workflow-trace / agent-workflow-panel 未被修改）
  7. TypeScript 编译检查：所有新增和修改文件通过 typecheck
